# RapidCull Operations Guide

This guide covers environment configuration, supported media formats, network access, backup and restore procedures, database/filesystem consistency checks, troubleshooting, and log locations for RapidCull operators.

---

## Environment Variables

RapidCull reads two environment variables at startup to configure where state is stored.

### `RAPIDCULL_DB_PATH`

Path to the SQLite database file. When set, the API server uses this path for all database operations.

- Set automatically when using `rapidcull start --db <path>`. The CLI resolves the path to an absolute form before passing it to the server process.
- Can also be set manually before starting the server with uvicorn directly:
  ```bash
  export RAPIDCULL_DB_PATH=~/photos/rapidcull.db
  uvicorn rapidcull.api:app --host 127.0.0.1 --port 8000
  ```
- If unset, the API starts without a database and image/gallery/person/trash endpoints are unavailable.

### `RAPIDCULL_LIBRARY_ROOT`

Root directory for the photo library. Used by the job executor for operations that produce or read derived assets (proxies, thumbnails).

- Set automatically when using `rapidcull start --library-root <path>`.
- Can be set manually alongside `RAPIDCULL_DB_PATH` when running uvicorn directly.
- If unset, library-root-dependent job operations will not have a configured root.

## Supported Media Formats

RapidCull discovers and processes files with the following extensions (case-insensitive):

| Category | Extensions |
|----------|-----------|
| JPEG | `.jpg`, `.jpeg` |
| PNG | `.png` |
| HEIC | `.heic` |
| RAW | `.cr2`, `.nef`, `.arw`, `.dng` |
| Video | `.mp4`, `.mov`, `.avi`, `.mkv`, `.webm` |

Files with any other extension are silently skipped during discovery. The `process-new` command reports skipped counts in its summary output.

## LAN / Network Access

By default the server binds to `127.0.0.1` (localhost only). To expose the API on a local network:

```bash
rapidcull start --host 0.0.0.0 --port 8000 \
  --db ~/photos/rapidcull.db \
  --library-root ~/photos
```

**Security notice**: RapidCull has no authentication or authorisation layer. It is designed for trusted LAN use only. Do not expose the server to the public internet without placing an authenticating reverse proxy (e.g. nginx with HTTP basic auth) in front of it.

**Firewall**: If your system runs a firewall (ufw, firewalld, iptables), open the port only to trusted subnets:

```bash
# Example: allow LAN subnet 192.168.1.0/24 on ufw
sudo ufw allow from 192.168.1.0/24 to any port 8000
```

## Backup & Restore

### What is backed up

`rapidcull backup` copies the SQLite database and any gallery JSON state files from an optional galleries root directory into a timestamped subdirectory under the backup root. Proxy/thumbnail files are not backed up (they are derived and can be regenerated by re-processing).

### Backup

```bash
rapidcull backup \
  --db ~/photos/rapidcull.db \
  --backup-root ~/photos/backups
```

To also back up gallery JSON state:

```bash
rapidcull backup \
  --db ~/photos/rapidcull.db \
  --galleries-root ~/photos/galleries \
  --backup-root ~/photos/backups
```

The command prints the number of files backed up, total bytes, and the path of the timestamped backup directory created.

### Restore

Restore overwrites the target database (and optionally the galleries root). The `--confirmed` flag is required to prevent accidental overwrites.

```bash
rapidcull restore \
  --backup-path ~/photos/backups/backup-2024-06-15T10-00-00 \
  --db ~/photos/rapidcull.db \
  --confirmed
```

To also restore gallery JSON state:

```bash
rapidcull restore \
  --backup-path ~/photos/backups/backup-2024-06-15T10-00-00 \
  --db ~/photos/rapidcull.db \
  --galleries-root ~/photos/galleries \
  --confirmed
```

Omitting `--confirmed` causes the command to exit with an error and make no changes.

### Backup rotation

RapidCull does not rotate or prune old backups automatically. Manage retention with a cron job or manual cleanup:

```bash
# Delete backups older than 30 days
find ~/photos/backups -maxdepth 1 -type d -mtime +30 -exec rm -rf {} +
```

## DB/Filesystem Consistency

The database records metadata about media files on disk. After a crash, manual file moves, or external deletions, the database and filesystem can diverge. The `check` command detects and optionally repairs this drift.

### When to run

- After an unclean server shutdown or system crash
- After manually moving, renaming, or deleting files outside of RapidCull
- Periodically as a scheduled maintenance task

### Check only (no changes made)

```bash
rapidcull check --db ~/photos/rapidcull.db
```

Output includes the check timestamp, total issues found, and a per-issue listing showing the issue kind, item ID, and detail message.

### Check with optional trash directory

If you use a trash directory, pass it so the check can account for items moved there:

```bash
rapidcull check --db ~/photos/rapidcull.db --trash-dir ~/photos/.trash
```

### Repair

Repair requires both `--repair` and `--confirmed`. Without `--confirmed` the command exits with an error.

```bash
rapidcull check \
  --db ~/photos/rapidcull.db \
  --repair \
  --confirmed
```

With a trash directory:

```bash
rapidcull check \
  --db ~/photos/rapidcull.db \
  --trash-dir ~/photos/.trash \
  --repair \
  --confirmed
```

Output reports the number of issues fixed and the number that could not be repaired.

## Troubleshooting

### "rapidcull: command not found"

The virtual environment is not activated and `.venv/bin` is not in `PATH`.

Fix — activate the venv, then retry:

```bash
source .venv/bin/activate
rapidcull start ...
```

Or call the binary directly without activating:

```bash
.venv/bin/rapidcull start ...
```

### "Error: port 8000 already in use"

Another process is already listening on port 8000, or a previous RapidCull instance is still running.

Fix — stop the existing instance:

```bash
rapidcull stop
```

Or start on a different port:

```bash
rapidcull start --port 8001 --db ~/photos/rapidcull.db --library-root ~/photos
```

### No images appear after ingest

Possible causes:

1. `--source-dir` pointed at the wrong directory, or the directory was empty.
2. Files have extensions not in the supported formats list (see [Supported Media Formats](#supported-media-formats)).
3. All files were already ingested on a previous run and were reported as skipped.

Check the `process-new` summary output: `Processed: N | Skipped: N | Failed: N`. If Skipped is non-zero and Processed is zero, the files were already recorded. If both are zero, no supported files were found.

### Face detection is very slow

This is expected behaviour. The InsightFace model loads from disk on the first face detection job and requires significant CPU (or GPU) time per image. Face detection runs as a background job; the API remains responsive while it processes. Progress can be monitored via the jobs endpoint.

### "exiftool: command not found" or ExifTool errors during ingest

ExifTool is not installed or is not on `PATH`.

Fix:

```bash
sudo apt-get install -y libimage-exiftool-perl
```

Verify after install:

```bash
exiftool -ver
```

### "ImageMagick cannot process HEIC/HEIF files"

ImageMagick is installed but was built without HEIF delegate support.

Fix:

```bash
sudo apt-get install -y libheif1 imagemagick
```

If the package is already installed, your ImageMagick build may not include HEIF support. In that case, install `libheif-dev` and rebuild or use a PPA that ships a HEIF-capable build. Verify support with:

```bash
magick -list format | grep -i heic
```

### "rawtherapee-cli: command not found" (RAW processing fails)

RawTherapee CLI is not installed.

Fix:

```bash
sudo apt-get install -y rawtherapee
```

Pass `--no-raw-pipeline` to `process-new` to skip RAW processing if RawTherapee is unavailable:

```bash
rapidcull process-new --source-dir /media/card --no-raw-pipeline
```

### Stale PID file after crash

If the server process was killed abruptly (kill -9, OOM killer, power loss), the PID file at `~/.local/run/rapidcull/rapidcull.pid` may remain.

`rapidcull stop` handles this automatically: if the recorded PID is no longer alive, it removes the stale PID file and reports a warning. No manual cleanup is needed.

If you want to clear it manually:

```bash
rm ~/.local/run/rapidcull/rapidcull.pid
```

## Log Locations

RapidCull does not write logs to a file by default.

**When started with `rapidcull start`**: the server subprocess is launched with stdout and stderr redirected to `/dev/null`. Server output is not captured. To retain logs, start the server manually with uvicorn and redirect output:

```bash
export RAPIDCULL_DB_PATH=~/photos/rapidcull.db
export RAPIDCULL_LIBRARY_ROOT=~/photos
nohup .venv/bin/uvicorn rapidcull.api:app \
  --host 127.0.0.1 --port 8000 \
  >> ~/photos/rapidcull.log 2>&1 &
echo $! > ~/.local/run/rapidcull/rapidcull.pid
```

**When started interactively** (foreground uvicorn): logs go to the terminal's stdout/stderr. Pipe or redirect as needed.

**CLI command output** (`process-new`, `backup`, `restore`, `check`): written to stdout (normal output) and stderr (errors). Redirect as needed:

```bash
rapidcull process-new --source-dir /media/card >> ~/photos/ingest.log 2>&1
```

**PID file location**: `~/.local/run/rapidcull/rapidcull.pid`
